# 认知提纯系统 - 需求文档

## 一、项目概述

一套个人认知管理系统，核心逻辑：用户只负责写入原始记录，AI自动提纯出复盘和心得，用户做终审后存入数据库。

设计哲学：**用户只写一次，AI做提纯，用户做质检。**

---

## 二、系统架构

### 技术栈

- 后端：Python（FastAPI 或 Flask）
- 数据库：SQLite（单用户，轻量，后期可迁移 PostgreSQL）
- AI：Anthropic Claude API（模型：claude-opus-4-6）
- 前端：移动端优先（后续可考虑魔改开源笔记 APP 作为写入口）
- 部署：用户自有服务器

### 整体流程

```
随手记（碎片）──┐
                ├──→ AI整合成当天日记 → 用户确认/修改 → 存入日记表
写完整日记 ────┘
                                              │
                                              ▼
                                    AI提纯 → 复盘（带标签）
                                              │
                                              ▼ 用户终审/修改
                                              │
                                              ▼
                                    AI提纯 → 心得（知 + 行）
                                              │
                                              ▼ 用户终审/修改
                                              │
                                              ▼
                                          存入数据库
```

## 三、数据库设计

### 3.1 收集箱表（inbox）

暂存区，碎片记录。每天整合进日记后标记为已处理。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增ID |
| content | TEXT NOT NULL | 碎片内容 |
| created_at | DATETIME | 记录时间 |
| status | TEXT DEFAULT 'pending' | pending / processed |
| processed_date | DATE | 被整合进哪天的日记 |

### 3.2 日记表（diary）

原始素材，一旦确认后不再修改。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增ID |
| date | DATE UNIQUE NOT NULL | 日期（主键级别，一天一条）|
| content | TEXT NOT NULL | 日记正文 |
| source | TEXT | manual（手写）/ merged（碎片整合）/ mixed（混合）|
| confirmed | BOOLEAN DEFAULT FALSE | 用户是否已确认 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后修改时间 |

### 3.3 复盘表（review）

从日记提纯而来，一篇日记可以生成多条复盘。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增ID |
| diary_id | INTEGER REFERENCES diary(id) | 关联日记 |
| event | TEXT | 事件描述 |
| decision | TEXT | 做了什么决策 |
| deviation | TEXT | 结果与预期的偏差 |
| attribution | TEXT | 归因分析 |
| tags | TEXT | 标签，逗号分隔（成功/失败/认知迭代/素材测试/等）|
| confirmed | BOOLEAN DEFAULT FALSE | 用户是否已终审 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后修改时间 |

### 3.4 心得表（insight）

从复盘提纯而来，分为"知"和"行"。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增ID |
| review_id | INTEGER REFERENCES review(id) | 关联复盘 |
| knowledge | TEXT | 知——学到的道理/认知 |
| action | TEXT | 行——SOP/可执行的操作标准 |
| related_insight_id | INTEGER REFERENCES insight(id) | 关联引用，指向被迭代的旧心得（认知版本链）|
| confirmed | BOOLEAN DEFAULT FALSE | 用户是否已终审 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 最后修改时间 |

## 四、API 接口设计

### 4.1 收集箱（写入口）

```
POST   /api/inbox              # 随手记一条碎片
GET    /api/inbox               # 获取今日待处理碎片
DELETE /api/inbox/{id}          # 删除某条碎片
```

### 4.2 日记

```
POST   /api/diary/merge         # 触发AI：将今日碎片 + 手写内容整合成日记
GET    /api/diary                # 获取日记列表（支持日期范围筛选）
GET    /api/diary/{date}         # 获取某天日记
PUT    /api/diary/{date}         # 用户修改日记
POST   /api/diary/{date}/confirm # 用户确认日记（锁定，触发AI提纯复盘）
```

### 4.3 复盘

```
GET    /api/reviews              # 获取复盘列表（支持标签筛选：?tag=成功 / ?tag=失败）
GET    /api/reviews/{id}         # 获取某条复盘详情（含关联日记）
PUT    /api/reviews/{id}         # 用户修改复盘
POST   /api/reviews/{id}/confirm # 用户确认复盘（触发AI提纯心得）
```

### 4.4 心得

```
GET    /api/insights             # 获取心得列表（支持筛选：?type=knowledge / ?type=action）
GET    /api/insights/{id}        # 获取某条心得详情（含关联复盘 + 迭代链）
GET    /api/insights/{id}/chain  # 获取某条心得的完整迭代链（从最早到最新）
PUT    /api/insights/{id}        # 用户修改心得
POST   /api/insights/{id}/confirm # 用户确认心得
```

### 4.5 数据库全局视图

```
GET    /api/dashboard            # 总览：今日状态、待终审数量、近期统计
GET    /api/trace/{diary_date}   # 溯源视图：某天日记 → 所有复盘 → 所有心得，三层完整脉络
```

## 五、AI 提纯逻辑

### 5.1 第一轮：碎片 → 日记（整合）

触发时机：用户调用 `/api/diary/merge`

输入：当天所有 pending 状态的碎片 + 用户可能手写的补充内容

AI任务：将碎片整合成一篇连贯的日记，保留原始信息，不添加AI自己的观点。

### 5.2 第二轮：日记 → 复盘（提纯）

触发时机：用户确认日记后自动触发

输入：确认后的日记正文

AI Prompt 要点：
- 从日记中提取所有可复盘的事件（一篇日记可能有多条）
- 每条复盘必须包含：事件、决策、结果偏差、归因
- 必须打标签：成功 / 失败 / 认知迭代（可多选）
- 归因必须具体，不能是泛泛的"应该多沟通"
- 如果日记中提到了对历史事件的看法改变，标记为"认知迭代"

AI 输出格式（JSON）：
```json
[
  {
    "event": "...",
    "decision": "...",
    "deviation": "...",
    "attribution": "...",
    "tags": ["成功", "素材测试"]
  }
]
```

### 5.3 第三轮：复盘 → 心得（提纯）

触发时机：用户确认复盘后自动触发

输入：确认后的复盘内容 + 历史心得库（用于检测是否存在认知迭代）

AI Prompt 要点：
- "知"：提炼出大方向的道理/认知，但必须跟具体事件挂钩，不能是空洞的正确废话
- "行"：必须是可执行的SOP，具体到"下次遇到X情况，执行Y步骤"，不能是"下次应该注意"
- 检测历史心得库中是否有被迭代的旧心得，如有则填写 related_insight_id
- 如果从复盘中提取不出有效的"行"（SOP），则该字段留空，不要硬编

AI 输出格式（JSON）：
```json
{
  "knowledge": "...",
  "action": "...",
  "related_insight_id": null
}
```

## 六、核心业务规则

1. **空白日允许存在。** 如果某天没有记录，系统中不会有该日的任何数据。不设废料层。
2. **日记确认后原则上不再修改。** 它是原始素材，保留当时的真实认知。
3. **复盘和心得可以在终审时修改。** 用户可以修正AI的归因错误、补充SOP细节、调整标签。
4. **认知迭代链：** 当新心得迭代了旧心得时，旧心得不删除，新心得通过 related_insight_id 指向旧心得，形成版本链。
5. **三层溯源：** 从任何一条心得出发，都能追溯到对应的复盘和原始日记。反过来也可以。
6. **标签筛选：** 用户可以在复盘表中按标签筛选，直接看到所有"成功"或所有"失败"的复盘。

## 七、前端架构

### 7.0 双端策略

| 端 | P0 方案 | P1 方案 |
|------|---------|---------|
| 电脑 | Web 应用（浏览器访问） | 同左 |
| 手机 | PWA（网页安装到桌面，接近原生体验） | 原生 APP（React Native 或 Flutter） |

P0 阶段只维护一套 Web 代码。前端框架推荐 React + TailwindCSS，同时配置 PWA（manifest.json + Service Worker），让手机用户可以"添加到主屏幕"获得类 APP 体验。

PWA 要求：
- manifest.json：应用名称、图标、主题色、启动URL、display: standalone
- Service Worker：缓存静态资源，支持离线打开（写入操作仍需联网）
- 移动端适配：触控友好的输入框、大按钮、底部导航栏

### 7.1 页面结构

#### 底部导航栏（移动端）/ 侧边导航（桌面端）

四个入口：写入 | 日记 | 数据库 | 设置

#### 写入页（首页，最高频操作）

- 默认打开就是输入框 + 发送按钮（随手记模式，摩擦力最低）
- 顶部切换：随手记 / 写日记
- 随手记模式：单行输入框，回车即发送，碎片立刻进收集箱
- 写日记模式：多行编辑器，支持较长文本输入，保存为当天日记
- 当天已有碎片时，底部展示碎片列表，可删除单条
- 一键"整合成日记"按钮：调用AI将碎片合并成日记草稿

#### 日记页

- 按日期倒序展示日记列表（卡片形式，显示日期 + 正文摘要 + 状态标签）
- 状态标签：草稿 / 已确认 / 已提纯
- 点击进入某天日记详情：
  - 查看完整内容
  - 编辑按钮（草稿状态可编辑）
  - 确认按钮 → 显示"AI提纯中..." → 完成后自动跳转到复盘终审页
- 空白日不显示，不占位

#### 复盘终审页

- 从日记确认后自动进入
- 也可从数据库页面进入
- 展示AI生成的复盘条目（可能多条）
- 每条复盘卡片包含：
  - 事件（可编辑）
  - 决策（可编辑）
  - 结果偏差（可编辑）
  - 归因（可编辑）
  - 标签选择器（成功 / 失败 / 认知迭代 / 自定义，可多选）
- 可删除AI生成的无效条目
- 可手动添加AI遗漏的复盘条目
- 确认按钮 → 触发AI提纯心得 → 跳转心得终审页

#### 心得终审页

- 展示AI生成的心得
- 每条心得包含：
  - 知（可编辑的文本区域）
  - 行/SOP（可编辑的文本区域，可为空）
  - 如果有关联引用：显示"迭代自：[旧心得摘要 + 日期]"，可点击跳转
- 确认按钮 → 存入数据库 → 显示"已入库"

#### 数据库浏览页

三个 Tab 页签：

**Tab 1：复盘库**
- 顶部标签筛选栏：全部 | 成功 | 失败 | 认知迭代（点击切换，高亮当前选中）
- 卡片列表，每条显示：事件摘要 + 标签 + 日期
- 点击展开：完整复盘内容 + 关联的日记原文 + 对应的心得
- 支持搜索

**Tab 2：心得库（知）**
- 所有"知"的列表
- 每条显示：知的内容摘要 + 来源日期
- 如果有迭代链，显示版本标记（如"v2 ← v1"）
- 点击展开：完整内容 + 关联复盘 + 关联日记

**Tab 3：心得库（行/SOP）**
- 所有"行"（SOP）的列表
- 每条显示：SOP内容摘要 + 来源日期
- 点击展开：完整内容 + 关联复盘 + 触发场景

#### 溯源视图

- 从数据库中任意一条记录点击"查看脉络"进入
- 纵向展示三层链路：日记原文 → 复盘分析 → 心得提炼
- 每层可折叠/展开
- 如果心得有迭代链，额外显示迭代历史时间线

#### 设置页

- API Key 配置
- 数据导出（JSON / CSV）
- 标签管理（增删自定义标签）
- 关于 / 版本信息

## 八、部署要求

- 用户自有服务器（Linux）
- 所有数据存储在本地 SQLite，不依赖第三方服务（除 Claude API）
- API密钥通过环境变量配置：`ANTHROPIC_API_KEY`
- 后端通过 HTTPS 暴露 API（建议 Nginx 反向代理 + Let's Encrypt 证书）
- 前端构建为静态文件，由 Nginx 同时托管
- Docker 部署（可选）：提供 Dockerfile 和 docker-compose.yml，一键启动

## 九、开发优先级

### P0（第一版必须有）

- 后端 API 全部接口
- SQLite 数据库四张表
- AI 提纯三轮逻辑（碎片→日记、日记→复盘、复盘→心得）
- Web 前端全部页面（React + TailwindCSS）
- PWA 配置（manifest + Service Worker，手机可安装到桌面）
- 收集箱随手记
- 日记整合与确认
- 复盘终审（带标签筛选）
- 心得终审（知 + 行）
- 数据库浏览（三个Tab + 标签筛选）
- 三层溯源视图

### P1（第二版）

- 原生 APP（React Native 或 Flutter，复用后端 API）
- 认知迭代链可视化（时间线形式）
- 全文搜索（日记 + 复盘 + 心得）
- 数据导出（JSON / CSV）
- 统计面板（每周/每月复盘数量、成功/失败比例、心得产出量）
- 离线写入（PWA 缓存碎片，联网后自动同步）

### P2（后续扩展）

- 微信 / Telegram bot 作为额外写入口
- 多标签自定义管理系统
- 任务栏模块
- 书架模块（阅读记录 + 读书笔记关联心得）
- 人脉模块（关键人物信息 + 关联复盘）
- 灵感池模块（独立于日记的想法孵化区）
